3. AWS SERVICES INTEGRATION
================================================================================


AWS BEDROCK (Claude AI) - CRITICAL SERVICE:
- Purpose: Intelligent conversational AI for all user interactions
- Model: Claude 3 Sonnet (Anthropic)
- Implementation: Context-aware responses for farmer/student/startup
- Features:
  * Maintains conversation history (last 10 messages)
  * User-type specific responses
  * Fallback to intelligent mock when unavailable
- Cost Optimization: DynamoDB caching of LLM responses
- Region: us-east-1
- Status: Integration ready, awaiting Bedrock access approval


AWS DYNAMODB:
- Purpose: NoSQL database for flexible schema data
- Use Cases:
  * Conversation history storage
  * LLM response caching (cost optimization)
  * Usage tracking and analytics
  * Session management
- Tables:
  * ChatMessages (conversationId, messageId, role, content, timestamp)
  * LLMCache (promptHash, userType, response, timestamp)
  * UsageMetrics (userId, action, metadata, timestamp)
- Benefits: Serverless, auto-scaling, low latency


AWS S3:
- Purpose: Object storage for user-uploaded files
- Use Cases:
  * Crop disease images (farmers)
  * Document uploads (students, startups)
  * Profile pictures
  * Generated reports
- Features:
  * Pre-signed URLs for secure uploads
  * Lifecycle policies for cost optimization
  * CloudFront integration for CDN
- Bucket Structure: samarthbharat-uploads/{userType}/{userId}/


AWS EC2:
- Purpose: Backend API server deployment
- Configuration:
  * Instance Type: t3.medium (2 vCPU, 4GB RAM)
  * Auto-scaling group (min: 1, max: 5)
  * Application Load Balancer
  * Health checks every 30 seconds
- OS: Amazon Linux 2
- Security: VPC, Security Groups, IAM roles


AWS RDS (PostgreSQL):
- Purpose: Relational database for structured data
- Use Cases:
  * User profiles and authentication
  * Transaction history
  * Scheme applications
  * Audit logs
- Configuration:
  * Engine: PostgreSQL 15
  * Instance: db.t3.micro
  * Multi-AZ for high availability
  * Automated backups (7-day retention)


AWS ELASTICACHE (Redis):
- Purpose: In-memory caching and session management
- Use Cases:
  * Rate limiting (100 requests/min per user)
  * Session storage
  * API response caching
  * Weather data caching (24-hour TTL)
  * Mandi price caching (1-hour TTL)
- Configuration: cache.t3.micro, Redis 7.0


AWS CLOUDFRONT:
- Purpose: Content Delivery Network (CDN)
- Use Cases:

AWS CLOUDWATCH:

4. AI/ML IMPLEMENTATION
================================================================================


CLAUDE AI (AWS BEDROCK) INTEGRATION:


Implementation File: backend/src/services/bedrock.ts
Key Features:
1. Context-Aware Responses:
   - Farmer: Agricultural terminology, mandi prices, crop advice
   - Student: Educational resources, exam preparation, scholarships
   - Startup: Business terminology, funding, compliance


2. Conversation Context Management:
   - Maintains last 10 messages for context window
   - In-memory storage during session
   - DynamoDB persistence for history
   - Automatic context pruning to manage token limits


3. Cost Optimization Strategy:
   - DynamoDB caching: Hash(prompt + userType) → cached response
   - Cache hit rate target: 30-40%
   - Estimated cost savings: $3-5 per 1000 requests
   - Cache TTL: 24 hours for general queries


4. Fallback Intelligence:
   - Smart mock responses when Bedrock unavailable
   - Context-aware fallbacks by user type
   - Graceful degradation without user disruption


CLAUDE VISION (Image Analysis):


Implementation File: backend/src/routes/chat.routes.ts
Use Case: Crop Disease Diagnosis
Process:
1. User uploads crop image (JPEG/PNG, max 5MB)
2. Image stored in S3 with pre-signed URL
3. Claude Vision API analyzes image
4. Returns:
   - Disease name (e.g., "Late Blight")
   - Severity level (Low/Medium/High)
   - Confidence score (0-100%)
   - Treatment recommendations
   - Preventive measures
5. Response cached in DynamoDB


Example Response:
{
  "disease": "Late Blight (Phytophthora infestans)",
  "severity": "High",
  "confidence": 87,
  "treatment": [
    "Apply copper-based fungicide immediately",
    "Remove infected leaves",
    "Improve air circulation"
  ],
  "prevention": [
    "Avoid overhead watering",
    "Plant resistant varieties",
    "Rotate crops annually"
  ]
}

7. API ENDPOINTS & FUNCTIONALITY
================================================================================


AUTHENTICATION ENDPOINTS (4):


POST /api/auth/request-otp
- Purpose: Send OTP to user's phone
- Input: { phoneNumber: string, userType: string }
- Process: Generate 6-digit OTP → Send via Twilio SMS
- Output: { success: boolean, message: string }
- Status: Twilio integration ready


POST /api/auth/verify-otp
- Purpose: Verify OTP and create session
- Input: { phoneNumber: string, otp: string }
- Process: Validate OTP → Generate JWT token
- Output: { token: string, refreshToken: string, user: object }
- Status: JWT implementation complete


POST /api/auth/refresh
- Purpose: Refresh expired JWT token
- Input: { refreshToken: string }
- Process: Validate refresh token → Generate new JWT
- Output: { token: string }
- Status: Complete


POST /api/auth/logout
- Purpose: Invalidate user session
- Input: { token: string }
- Process: Blacklist token in Redis
- Output: { success: boolean }
- Status: Complete


CHAT ENDPOINTS (2):


POST /api/chat/message
- Purpose: Send message and get AI response
- Input: { 
    message: string, 
    userType: string, 
    conversationId: string,
    attachments: array 
  }
- Process:
  1. Check DynamoDB cache for similar query
  2. If cache miss, call AWS Bedrock (Claude AI)
  3. Cache response in DynamoDB
  4. Save message to conversation history
  5. Track usage metrics
- Output: {
    conversationId: string,
    response: string,
    suggestions: array,
    quickActions: array
  }
- Status: Bedrock integration ready, mock responses working


GET /api/chat/history
- Purpose: Retrieve conversation history
- Input: { conversationId: string }
- Process: Fetch from DynamoDB
- Output: { conversations: array, total: number }
- Status: Complete


GOVERNMENT SCHEMES ENDPOINTS (3):


GET /api/schemes
- Purpose: List government schemes with filters
- Query Params: 
  * userType: farmer|student|startup
  * category: agriculture|education|business
  * keyword: string
- Process: Filter mockSchemes.json (12 schemes)
- Output: { schemes: array, total: number }
- Status: Complete with mock data


GET /api/schemes/:id
- Purpose: Get detailed scheme information
- Input: Scheme ID (e.g., "pm-kisan")
- Output: {
    id, name, nameLocal, description, category,
    targetUserTypes, eligibility, benefits,
    applicationProcess, requiredDocuments,
    officialLink, helplineNumber, isOngoing
  }
- Status: Complete


POST /api/schemes/check-eligibility
- Purpose: Check user eligibility for scheme
- Input: { schemeId: string, userProfile: object }
- Process: Match user profile against eligibility criteria
- Output: { eligible: boolean, reasons: array }
- Status: Complete
